This page is about a very ordinary backend moment: the app is mostly fine, but
one part of it got hot.

A parser. A validator. A hash. A render. A compression step. The profiler points
at a few functions, the event loop starts to feel sticky, and now the fix has to
have a shape.

If you want the machinery, read [Architecture](/extras/architecture/). This page
is about the feeling behind Knitting: the problem is often smaller than the
infrastructure we reach for.

## The hot-function problem

CPU pressure in a JavaScript app is rarely spread evenly. In real codebases, a
small set of functions often creates most of the cost. It is not a law of
nature, just a familiar shape: the app has thousands of functions, but the heat
lives in five of them.

That matters because it changes the size of the fix. You may not have an
application problem. You may have a function problem. The usual escape routes
are all bigger than that: either change nothing structural, or restructure far
more of the system than the hot path deserves.

## The three escape routes

When the main thread starts paying for those functions, JavaScript developers
usually pick one of three doors.

**Door one: stay in-process and eat the cost.** This is the default, and for a
while it works. But every millisecond a hot function spends on the main thread
is a millisecond your server is less available for everything else. Cheap routes
queue behind expensive ones. Timers drift. Scaling out with more processes or
containers dilutes the pain, but it does not remove the hot path from the event
loop.

**Door two: hand-roll workers.** This is closer to the right shape, and it is
also why many people bounce off workers. The runtime gives you `postMessage`;
the rest is yours: request IDs, routing, error marshalling, promise wiring,
lifecycle management, payload choices. After all of that, the per-message cost
can still make small tasks feel silly to offload. That plumbing is the original
reason Knitting exists.

**Door three: promote the function into a microservice.** This door is
well-paved, so teams take it. Sometimes they should. Separate ownership,
separate deploy cadence, separate failure domain: those are real reasons to
cross the network. But for one hot function owned by the same team in the same
repo, the bill is strange: a network hop, JSON on both sides, service discovery,
deployment, dashboards, an on-call surface. The function did not ask for a
hostname. It asked for somewhere else to run.

## The fourth shape

Knitting's position is that there is a missing option between "function call"
and "service boundary":

> Keep the code local. Move the execution boundary.

The hot function stays in your repo, in your language, in your types, one
`import` away from its caller. What changes is *where it executes* and *what it
is allowed to do there*. Export it, hand it to a pool, and call it like the
async function it already was. Underneath, it runs on a real thread or an
isolated process behind the boundary you chose.

That is the sense in which Knitting is a **function-level execution boundary**,
not just a worker helper. Compare what each door costs for the same five hot
functions:

| | In-process | Hand-rolled workers | Microservice | Knitting |
|---|---|---|---|---|
| Main thread protected | no | yes | yes | yes |
| Code stays in the app | yes | mostly | no | yes |
| Call-site ergonomics | function call | protocol you wrote | HTTP client | function call |
| Transport cost | none | `postMessage` per call | network + JSON | shared-memory mailboxes |
| Isolation available | none | thread only | full, always-on | a dial, per pool |
| New deploy unit | no | no | yes | no |

Two of those cells are the heart of the thing.

**Transport cost** is what historically made workers disappointing for small
tasks: sometimes reaching the worker costs more than the work. Knitting uses
shared-memory mailboxes instead of the runtime's message queue, which keeps
small and medium calls economical. The [Architecture](/extras/architecture/)
page explains the mechanism, and the [benchmarks](/benchmarks/introduction/)
show the resulting shape honestly, including where it does not win.

**Isolation is a dial, not an always-on tax.** A microservice gives you process
isolation whether or not you need it, and bills every call for the privilege.
Knitting stacks [four independent safety layers](/extras/architecture/#safety-layers):
in-process guards, a bootstrap hook, runtime-native permissions, and a real
OS-sandboxed process. You pick the rung each pool deserves. Trusted math can
stay on cheap threads; an untrusted plugin can run behind `bwrap`, with
[`importTask`](/guides/defining-tasks/#module-loading) keeping its code off the
host entirely.

To be clear about what this argument is *not*: Knitting does not replace
distributed systems. If you need cross-machine scale, separate failure domains,
or team-level ownership boundaries, you need the network. Knitting is built to
sit *inside* each node of that system, not instead of it. What it removes is the
pressure to create a distributed system for the wrong reason: just to escape the
main thread.

> Not every hot path deserves a hostname.

## Where it's going

:::note[Direction, not commitment]
Everything below is roadmap as of June 2026. None of it is shipped API, and the shape may change or be cut as the design matures. Build against what the [guides](/guides/defining-tasks/) document today.
:::

Knitting today is task-call oriented: one request in, one response out, over
shared memory. The foundation underneath that -- mailboxes, payload buffers,
named mappings that cross process boundaries -- is more general than the
request/response API on top of it. Two extensions follow naturally from there.

**Channels (nearer term).** Some same-host communication is not a task call:
progress streams, long-lived coordination between lanes, producer/consumer
flows. Today the honest answer for those is `MessagePort`. A channel primitive
on the shared-memory transport would cover the cases where a task call is the
wrong shape but the transport's economics still matter.

**Cross-language, same-host (longer term).** The mailbox protocol and shared
mappings are not JavaScript-specific. A process worker already reaches memory
through a named mapping rather than anything runtime-magical. The longer-term
prize is letting JavaScript hand bytes to another language's runtime on the same
machine without re-marshalling the payload at every hop. Large payloads staying
in place while ownership moves is the same idea Knitting is built on; extending
it across language boundaries is the natural end of that road.

Those ideas are part of why the transport layer is engineered more carefully
than a worker pool strictly needs. Until they ship, though, judge Knitting on
what it does now.
